# RFC-0001: Runtime 对外交互层选型（MCP / Agent / Skill / Plugin / A2A 的分层定位）

> Status: Draft
>
> Last Updated: 2026-07-02
>
> Owners: @tangjianfang
>
> Relates-To: PS-0001

---

## 1. 动机

`PS-0001` 指出 Windows 平台 AI 开发工具在细分领域（USB/HID 等外设通信、音视频、GUI）缺乏深度模块化。
在讨论"本项目应以 Skill / Agent / Plugin / MCP 中哪种方式对外交互"时
（见 `99_Archive/Discussion/2026-07-02_AI交互方式选型讨论.md`），发现若将项目简单等同于四者中的任意一个，
会直接违反 `Project_Charter.md` 的项目边界（不是另一个 AI Chat / Claude Code / Cursor / 代码生成器）与
`README.md` Principle 5（Provider Independent）。因此需要一份可追溯的架构方案，明确这四种机制在 Runtime 中
各自的定位与边界，作为 Phase 3 Architecture 阶段的起点。

## 2. 方案概述

Skill / Agent / Plugin / MCP 四者回答的是不同层面的问题，不是互斥的单选项：

| 概念 | 回答的问题 | 本质 |
|---|---|---|
| MCP | 外部 AI Host 怎么发现并调用本项目的能力 | 标准化、厂商中立的协议层 |
| Agent | 目标怎么被自主拆解、执行、验证 | Runtime 内部执行循环 |
| Skill | 领域知识/流程怎么打包、按需加载 | 可插拔的知识/流程包 |
| Plugin | 第三方怎么给 Runtime 新增 Capability 实现 | SDK 扩展机制 |

核心身份保持不变：Provider-independent 的 Capability + Verification Engineering Kernel/Runtime。
Skill/Agent/Plugin/MCP 是从这个稳定核心向外延伸出的四类前端/边界机制，不应反过来定义项目身份
（类比 Capability First 原则：Capability 稳定，底层 Tool 可替换）。

## 3. 详细设计

按 `04_Runtime/README.md` 现有模块（Kernel / Context / Capability / Workflow / Agent / Memory / EventBus /
Provider）分层落位：

1. **对外暴露层 → MCP**（对应 Provider 边界之外的宿主接入面）
   - Runtime 作为 MCP Server，将 Capability 封装为 MCP Tools（可能还有 Resources/Prompts）供外部 AI Host
     （Claude Code / Cursor / 其它兼容 MCP 的工具）发现与调用。
   - 这是四者中唯一天然满足 Provider Independent 的机制：一次实现可被任意兼容 MCP 的 Host 复用，避免为
     每个 AI 工具单独集成造成的 M×N 问题（参考 `02_Research/MCP/README.md`）。

2. **内部执行层 → Agent（`04_Runtime/Agent/`，已有占位目录）**
   - Goal First 原则要求 Runtime 接收的是 Goal 而非 Prompt，因此必须有感知-规划-行动-观察的自主循环，
     负责把 Goal 拆解为对 Capability/Workflow 的调用序列，并观察执行结果决定下一步。
   - 这一层是必要的内部机制，不是可选的"交互范式"，因此不与 MCP/Skill/Plugin 处于同一选择维度。

3. **知识复用层 → Skill（`02_Research/Skill/README.md`）**
   - Skill 提供"怎么做"的结构化领域知识（如 USB/HID 协议规范），与 Capability 提供"能验证的执行"互补，
     而非重叠或替代关系。
   - 直接呼应 `PS-0001` 的核心痛点：细分领域知识可封装为独立 Skill 包，被本项目未来的内部 Agent，
     以及外部 Claude Code/Cursor 等工具按需加载复用，不因更换 AI Provider 而失效。
   - 具体封装格式与加载机制留待后续 Prototype 验证（见第 7 节未决问题）。

4. **扩展机制层 → Plugin SDK（`05_SDK/PluginSDK.md`，已有占位）**
   - 定位为第三方给 `Capability` 提供新的底层 Tool 实现的生态扩展通道，解决"生态扩展"问题，
     不参与"AI 交互范式"之争，因此不作为对外交互的主范式。

5. **A2A → 暂缓**
   - 当前阶段多 Agent 协作（如未来 Planner/Reviewer/Builder 角色）发生在单进程 Runtime 内部，
     不需要跨组织的 Agent Card 发现与 Task 生命周期状态机（参考 `02_Research/A2A/README.md`）。
   - 若未来出现跨进程/跨组织的 Agent 协作需求，再重新评估引入 A2A。

## 4. 备选方案

- **纯 MCP 方案**（把项目等同于一个 MCP Server）：被否决，因为无法回答"Goal 如何被拆解执行"
  （Agent 职责）与"细分领域知识如何封装复用"（Skill 职责），MCP 只解决外部调用发现问题。
- **纯 Agent 方案**（把项目等同于一个智能体应用）：被否决，因为缺少 Provider-neutral 的标准化对外接口，
  会退化为又一个类似 Claude Code/Cursor 的绑定型工具，违反 Project_Charter 边界。
- **纯 Skill 方案**（把项目等同于一组技能包）：被否决，因为 Skill 本身不执行、不产出可验证结果，
  无法满足 Verification First 原则。
- **纯 Plugin 方案**（把项目等同于某宿主的插件）：被否决，因为会被绑定在特定宿主的插件生态里，
  同样违反 Provider Independent 原则。
- **引入 A2A**：被否决（暂缓），因为当前不存在跨组织/跨框架的 Agent 协作场景，属于超前设计。

## 5. 影响与风险

- 对现有代码无影响：项目当前处于 Phase 0-2（Charter/Problem/Research），`04_Runtime` 尚无实质代码实现。
- 风险：在 Prototype 验证前过早锁定具体协议细节（如 MCP Tool Schema、Skill 封装格式）可能导致后续推翻成本
  增加；本 RFC 仅锁定分层定位，具体接口设计留待 Phase 4 Prototype 阶段验证后再形成 ADR。
- 风险：Skill/Plugin 的调研笔记（`02_Research/Skill/`、`02_Research/Plugin/`）目前主要参考 Anthropic/VS Code
  生态的既有实现，需要在后续设计中转化为 Provider-neutral 的通用范式，避免不自觉地与单一 Provider 绑定。

## 6. 验证方式

- 本 RFC 现阶段是分层定位的架构方案，暂无可执行验证；后续验证方式为 Phase 4 Prototype（EXP）阶段针对
  MCP Server 封装 Capability、Skill 包加载机制分别设计原型验证，验证通过后再固化为对应 ADR
  （参考 `ADR-0001` 同类先例：文档组织类决策暂无可执行验证，后续通过观察是否遵循约定来验证）。

## 7. 未决问题

- MCP Tool Schema 与 Capability 的对应关系如何设计（一个 Capability 对应一个 Tool？还是按 Workflow 粒度
  聚合）。
- Skill 封装格式如何定义，才能同时满足"可被本项目内部 Agent 使用"与"可被外部任意 Provider/Agent 复用"
  （Provider Independent）两个目标，而不是直接照搬某一家的私有 Skill 格式。
- Plugin SDK 与 Provider SDK 的边界如何划分（两者都涉及"第三方扩展"，但扩展的对象不同）。
- 以上三点留待下一次讨论或 Prototype 验证后再推进为独立 ADR。
