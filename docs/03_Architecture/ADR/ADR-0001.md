# ADR-0001: 04_Runtime 采用生长式目录（Growing Tree），不预先按 DDD 全量展开

> Status: Accepted
>
> Last Updated: 2026-07-02
>
> Owners: @tangjianfang
>
> Relates-To: PS-0001

---

## 1. 背景（Context）

`04_Runtime` 当前按模块（Module）组织：`Kernel/ Context/ Capability/ Workflow/ Agent/ Memory/ EventBus/ Provider/`。
讨论中指出：随着内容增长（例如 `Capability` 下会出现 Build / Git / USB / DXGI / WinDbg / Device 等数十个子领域，
`Workflow` 会出现 FixBug / ReviewCode / BuildProject 等模式，`Agent` 会出现 Planner / Reviewer / Builder 等角色），
`04_Runtime` 会迅速膨胀失控。

进一步观察：每一个 Capability（如 USB）本质上都是一个独立领域（Domain），未来需要自己的 ADR / RFC / Prototype /
SDK / Examples / Documentation，理想的远期形态是按 DDD（Domain Driven Design）组织整个 Runtime，
并为 `Workflow`、`Agent`、`Context`、`EventBus` 等设计更丰富的内部结构，同时增加 `Runtime/Common/` 承载跨领域共享概念
（State / Lifecycle / Identity / Metadata / Dependency / Version / Configuration）。

但现阶段 `04_Runtime/*` 所有子目录都仍是"待填充"的占位 README，尚无实质内容。

## 2. 决策（Decision）

`04_Runtime` 的目录结构采用**生长式目录（Growing Tree）**原则演进，不在当前阶段一次性按 DDD 目标形态展开：

1. **分阶段生长**：
   - 阶段一：`04_Runtime` 仅保留少数一级目录，每个对应一个核心概念（现状：Kernel / Context / Capability /
     Workflow / Agent / Memory / EventBus / Provider）。
   - 阶段二：当某个概念（如 `Capability`）内容积累到一定规模时，再拆分为子领域目录（如 Build / Git / USB / DXGI）。
   - 阶段三：当某个子领域（如 `USB`）继续成长时，再为其建立 ADR / RFC / Prototype / SDK / Examples 等完整结构。
2. **三篇文档门槛规则**：任何新建子目录（阶段二、阶段三的拆分），必须已经有至少三篇文档确定要放入该目录，
   否则不允许创建该目录。
3. **索引优先于目录**：任何可能持续膨胀的领域，其 `README.md` 必须维护一张索引表（至少包含名称 / Status /
   ADR / SDK / Prototype 等列），用索引承载可发现性，而不是提前建空目录。
4. **远期目标形态作为参考、非当前结构**：DDD 形态（`Kernel / Core / Capability / Workflow / Agent / Context /
   Provider / SDK / Common / RuntimeHost / Verification / Scheduler / EventBus`）记录于
   `99_Archive/Discussion/2026-07-02_Runtime目录组织方式.md`，作为未来演进方向参考，不作为当前必须实现的目录结构。

## 3. 理由（Rationale）

- 提前建立大量空目录会形成"空架子"（空目录森林），降低仓库可信度，且与项目"Phase 0-3 只做设计草稿，
  不实现 Runtime"的当前阶段纪律相悖。
- 目录应反映已经存在的知识密度，而不是预测的知识密度；"生长"符合仓库既有的 Draft → Review → Accepted 文档
  状态机思路，把同样的渐进原则应用到目录结构上。
- 三篇文档门槛提供一个简单、可执行、可长期遵守的量化标准，避免"何时该拆目录"变成主观判断。
- 索引表比目录层级更灵活：可以在不移动文件的情况下先记录规划中的领域及其状态，避免过早的物理拆分。

## 4. 后果（Consequences）

正面：
- 仓库结构与实际内容密度保持一致，五年后仍可保持整洁，不会积累半成品空目录。
- 新增领域有明确、一致的判断标准（三篇文档门槛），减少目录结构上的临时决策。

负面 / 需注意：
- 索引表需要人工维护，若不遵守更新纪律会失去作用，需要在后续 Workflow 或模板中提醒维护。
- 拆分时机判断仍需人工触发（三篇文档门槛只是必要条件，不是自动化机制）。

## 5. 验证（Verification）

- 本决策为文档组织约定，暂无可执行验证；后续验证方式为观察 `04_Runtime` 各子目录在阶段二 / 阶段三拆分时
  是否遵循三篇文档门槛与索引表约定。
- 若违反本决策（例如提前创建空的领域子目录），应在下一次讨论中记录并考虑是否需要 Superseded 本 ADR。
