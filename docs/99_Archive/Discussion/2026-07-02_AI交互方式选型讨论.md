# 讨论记录：Runtime 应以 Skill / Agent / Plugin / MCP 中哪种方式对外交互

> Date: 2026-07-02
>
> Owners: @tangjianfang
>
> 涉及主题：Skill / Agent / Plugin / MCP / A2A / Capability / Provider Independent / RFC-0001

---

## 1. 讨论起点

提问：当前项目更适合作为 Skill？智能体（Agent）？插件（Plugin）？MCP 交互？还是其它更好的 AI 交互方式？

## 2. 核心洞察

Skill / Agent / Plugin / MCP 四者**不是同一层面的互斥选项**，而是回答不同问题的机制，把项目"等于"其中
任何一个都会违反 `Project_Charter.md` 的项目边界（不是另一个 AI Chat/Claude Code/Cursor/代码生成器）与
`README.md` Principle 5（Provider Independent）：

| 概念 | 回答的问题 | 本质 |
|---|---|---|
| MCP | 外部 AI Host 怎么发现并调用本项目的能力 | 标准化、厂商中立的协议层 |
| Agent | 目标怎么被自主拆解、执行、验证 | Runtime 内部执行循环 |
| Skill | 领域知识/流程怎么打包、按需加载 | 可插拔的知识/流程包 |
| Plugin | 第三方怎么给 Runtime 新增 Capability 实现 | SDK 扩展机制 |

用 Capability First 原则类比：Capability（如 Build Project）稳定，底层 Tool 可换；同理，"Runtime 如何被
感知和调用"也应有稳定核心（Capability Kernel + Verification），Skill/Agent/Plugin/MCP 只是从核心延伸出的
前端，不应反过来定义项目身份。

## 3. 分层结论

1. **核心身份不变**：Provider-independent 的 Capability + Verification Engineering Kernel/Runtime。
2. **对外暴露层 → 优先 MCP**：四者中唯有 MCP 天然满足 Provider Independent，可避免为每个 AI 工具单独
   集成（M×N 问题）。
3. **内部执行层 → Agent Loop（非选择题）**：Goal First 原则要求接收 Goal 而非 Prompt，必然需要
   感知-规划-行动-观察的自主循环去拆解调度。
4. **知识复用层 → 引入 Skill，但作为知识包而非项目身份**：`PS-0001` 的核心痛点（USB/HID/音视频/GUI
   缺乏深度模块化）正是 Skill 机制的用武之地，与 Capability 互补（Skill 给"怎么做的知识"，Capability
   给"能验证的执行"）。
5. **扩展机制层 → Plugin SDK 保留**：定位为第三方给 Capability 提供新 Tool 实现的生态扩展通道，不参与
   "AI 交互范式"之争。
6. **A2A → 暂缓**：当前阶段多 Agent 协作发生在单进程内，不需要跨组织的 Agent Card 发现机制。
