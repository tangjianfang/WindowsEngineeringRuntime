# 讨论记录：04_Runtime 目录组织方式

> Date: 2026-07-02
>
> Owners: @tangjianfang
>
> 涉及主题：Claude Code Runtime / Windows Native / AI Runtime / Capability / Workflow / Agent / Context / EventBus / Project Charter

---

## 1. 讨论起点

当前 `04_Runtime` 按模块（Module）组织：

```text
04_Runtime/
  Kernel/
  Context/
  Capability/
  Workflow/
  Agent/
  Memory/
  EventBus/
```

问题：一年后，`Capability`（Build / Git / USB / DXGI / WinDbg / Device / FileSystem / Network / Compiler / Debugger ...）、
`Workflow`（FixBug / ReviewCode / BuildProject / DebugDevice / ReleaseFirmware ...）、
`Agent`（Planner / Reviewer / Builder / Debugger / Tester ...）都会持续膨胀，`04_Runtime` 会失控。

## 2. 核心洞察

**Capability 本身也是一种 Architecture**：每一个 Capability（例如 USB）未来都需要自己的 ADR / RFC / Prototype / SDK /
Examples / Test / Documentation，本质上是一个独立的领域（Domain），而不只是一个目录。

由此推广：`Workflow`、`Agent`、`Context`、`EventBus` 都可能演化出同样丰富的内部结构（Patterns、Design、Prompt、
Memory、Architecture 等子文档）。

## 3. 提出的目标形态（远期）

按 DDD（Domain Driven Design）组织 Runtime：

```text
Runtime/
  Kernel/
  Core/
  Capability/
  Workflow/
  Agent/
  Context/
  Provider/
  SDK/
  Common/
  RuntimeHost/
  Verification/
  Scheduler/
  EventBus/
```

以及每个领域内部的书籍式结构示例（如 `Capability/USB/README.md, Vision.md, Architecture.md, Lifecycle.md,
Capability.md, State.md, Workflow.md, SDK.md, Examples.md, Research.md, ADR/, RFC/, Prototype/`）。

`Runtime/Common/` 用于沉淀跨领域共享的概念（State / Lifecycle / Identity / Metadata / Dependency / Version /
Configuration），避免各领域重复定义。

## 4. 自我否定：现在不能这样做

上述形态是**未来**，不是**今天**。现阶段所有 `04_Runtime/*` 子目录都只有占位 README（"待填充"），
提前建好 100 个空目录只会形成"空架子"（空目录森林 / 文档坟场）。

## 5. 最终结论（已采纳，见 ADR-0001）

1. **生长式目录（Growing Tree）**：目录不是提前设计出来的，而是随着知识增长自然分裂出来的。
   - 第一阶段：`04_Runtime` 只有少数一级目录，每个对应一个核心概念。
   - 第二阶段：某个概念（如 Capability）内容足够多时，再拆成子领域（Build / Git / USB / DXGI ...）。
   - 第三阶段：某个子领域（如 USB）继续成长时，再拥有自己的 ADR / RFC / SDK / Prototype / Examples。
2. **三篇文档门槛规则**：任何新建子目录，必须已经有至少三篇文档需要放入，否则不允许创建。
3. **索引优先于目录**：比起提前建目录，更缺的是索引（Index）。每个可能膨胀的领域（如 Capability）应在其
   `README.md` 中维护一张索引表（Capability / Status / ADR / SDK / Prototype 等列），随内容增长登记，而不是
   靠目录结构本身承载信息。
