格式：作为 <角色>，我希望 <目标>，以便 <价值>。

## 工程目标表达

- 作为 Windows Native 工程师，我希望能直接向工具描述工程目标（如"支持新的 HID 指令"），
  以便不必自己把目标拆解成一步步的操作指令再喂给工具。

## 外设通信排查

- 作为负责设备驱动/外设通信的工程师，我希望有系统化的协议知识（如 USB 描述符、HID Report
  结构、Mass Storage 命令）可供查阅，以便在抓包排查问题时不用每次都重新翻查官方规范原文。

## 能力复用

- 作为团队中的工程师，我希望"如何构建/如何验证/如何抓包分析"这类工程能力是可复用、可组合的，
  以便新成员或新项目也能直接复用既有能力，而不用从零摸索。

## 结果可验证

- 作为工程负责人，我希望 AI 辅助产出的每一个结果都能通过 Build / Test / Benchmark / Device
  验证，以便对交付质量有可追溯的信心，而不是单纯依赖对 AI 输出的信任。

## Provider 独立

- 作为工具链维护者，我希望底层 AI Provider 可以按需替换（Claude / GPT / Gemini / 本地模型），
  以便不被单一厂商的价格、可用性或政策变化锁定。
