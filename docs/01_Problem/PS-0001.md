## 1. 背景

当前主流 AI 开发工具（Claude Code / Cursor / Codex CLI 等）大多面向通用软件开发场景设计，
对 Windows Native 工程中的细分领域（外设交互通信、音视频、GUI 等）缺乏针对性支持。

## 2. 用户是谁

面向 Windows 平台的开发者。

## 3. 痛点

目前的 AI 开发工具，在 Windows 平台下的很多细分领域，例如：

- 外设交互通信（USB / HID / Device 通信）
- 音视频领域（Audio / Video / DXGI 等）
- GUI（桌面应用界面）

缺乏深度的模块化支持，工具停留在通用代码生成层面，无法沉淀为可复用、可组合的工程能力。

## 4. 今天如何工作

<初稿，详见 `DailyWorkflow.md`（待与真实场景核对）。>

工程师遇到外设通信问题时，通常需要在多个分散来源（官方规范、厂商文档、抓包工具、调试器）之间
手动切换和核对信息；通用 AI 工具由于缺乏这些细分领域的结构化知识，难以提供有针对性的帮助；
排查过程中获得的经验也缺乏统一沉淀渠道，难以在团队内复用。

## 5. 未来希望怎样工作

<初稿，详见 `UserStories.md`（待与真实场景核对）。>

工程师可以向 Runtime 直接描述工程目标（如"支持新的 HID 指令"），Runtime 具备外设通信（USB/HID
等）协议的结构化知识作为 Capability 的知识底座，能够辅助进行抓包数据的协议解析与问题定位；
产出结果可通过 Build / Test / Benchmark / Device 验证；相关知识与工作流程可在团队与项目间复用，
不因更换 AI Provider 而失效。

## 6. 成功判据

<初稿，待与真实场景核对。>

- 工程师描述工程目标后，无需自行拆解成具体操作步骤即可推进任务。
- 外设通信协议知识（如 USB 各类规范）已结构化整理，可被查阅或未来被 Capability 复用，
  减少"每次现查现学"的重复劳动（参考 `06_Reference/USB/README.md`）。
- 关键工程结果均可通过既有验证手段（Build/Test/Benchmark/Device）确认，而非仅凭对 AI 输出的信任。
