## 1. 通用 AI 工具与 Windows Native 工程脱节

- 主流 AI 编程工具面向通用软件开发场景设计，对 Windows Native 工程里的细分领域
  （外设交互通信、音视频、GUI 等）缺乏针对性支持，工具停留在"生成代码片段"层面。

## 2. 细分领域知识无法沉淀为可复用能力

- 工程师在处理 USB/HID 等外设通信问题时积累的经验（协议字段含义、抓包排查方法）
  往往只存在于个人记忆或零散笔记中，无法沉淀为团队/工具可复用的结构化能力。
- 每次遇到新的外设通信问题，都要重新从官方规范、社区资料中现查现学，重复劳动。

## 3. 工具驱动导致工程流程脆弱

- 工程师围绕 MSBuild、CMake、调试器、设备工具等一系列离散工具展开工作，
  工具自身的变化（版本升级、行为调整）不断侵蚀工程流程的稳定性。
- 缺乏一个稳定的"能力抽象层"，导致底层工具变化时上层工作流程也要跟着改变。

## 4. AI 生成结果难以验证

- 通用 AI 工具生成的代码/建议缺乏与 Build / Test / Benchmark / Device 等工程验证手段的
  直接联动，工程师需要额外手动验证结果是否真的解决了问题。

## 5. Provider 锁定风险

- 多数现有 AI 工具与特定模型/Provider 强绑定，一旦更换 Provider，积累的项目级配置、
  记忆文件、工作流约定难以平滑迁移。
