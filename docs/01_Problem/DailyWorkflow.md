## 1. 遇到新的外设通信需求时

- 工程师先查阅设备厂商提供的规范/手册（如果有），或直接对照官方 USB-IF / Class 规范逐条核对；
  规范分散在多个来源，缺乏一个统一、结构化的知识入口。
- 借助抓包工具（如 Wireshark + USBPcap）观察实际总线流量，手动比对规范与抓包结果，
  逐字节还原描述符/命令含义；这一过程高度依赖个人经验，难以复用给团队其他成员。

## 2. 编写/调试代码时

- 使用 MSBuild / CMake 等构建工具，配合 WinDbg 等调试器逐步排查问题；工具之间缺乏统一的
  上下文传递机制，工程师需要在多个工具窗口间手动切换、复制粘贴信息。
- 借助通用 AI 编程助手生成代码片段，但助手对 Windows Native 细分领域（外设通信协议等）缺乏
  专门知识，给出的建议往往需要工程师自行大量修正。

## 3. 验证结果时

- 工程师手动运行 Build / Test / Benchmark，或将设备连接到真机进行验证；
  验证步骤本身没有被结构化记录，难以复现或自动化重跑。

## 4. 积累与传承经验时

- 排查过程中获得的协议知识、常见坑点，通常停留在个人笔记、聊天记录或口头传授中，
  缺乏沉淀到项目文档、供团队与工具共同复用的机制。
